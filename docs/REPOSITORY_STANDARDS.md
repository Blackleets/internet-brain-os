# Repository Standards

## Folder Structure
- `/apps`: Web applications (extension, dashboard, etc.). Each app is a separate package but not necessarily a workspace package for building.
- `/packages`: Shared libraries and core modules (kernel, obsidian, shared, skills, agents). Each is a workspace package with its own `package.json` and `tsconfig.json`.
- `/docs`: Documentation files (this folder).
- `/docs/ENGINEERING_STANDARDS.md`, architecture, protocols, etc.
- `/brain`: Institutional memory in Markdown format (for Obsidian sync).
- `/knowledge/agent-sessions`: Session logs for AI agents (Obsizon-compatible).
- `/prompts`: Prompt templates for AI agents.
- `/scripts`: Utility scripts (Python, Node.js, etc.).
- `/tasks`: Task definitions (e.g., `tasks/phase-0.md`).
- `/tests`: End-to-end or integration tests (if any).
- `/.github`: GitHub workflows, issue templates, pull request templates.

## Naming Conventions
- **Directories**: kebab-case (e.g., `agent-sessions`, `package-json`).
- **Files**: 
  - Markdown: `.md` (e.g., `README.md`, `CHANGELOG.md`).
  - TypeScript: `.ts` (or `.tsx` for React).
  - JSON: `.json` (e.g., `package.json`, `tsconfig.json`).
  - YAML: `.yaml` or `.yml` (e.g., `pnpm-workspace.yaml`).
  - Scripts: `.py` for Python, `.js` or `.mjs` for Node.js, `.sh` for shell.
- **Branches**: 
  - `main`: Production-ready, deployable state.
  - `phase/<major>.<minor>`: For phased development (e.g., `phase/0.1-create-technical-skeleton`).
  - `feature/<short-description>`: For feature branches (to be used in Phase 0.2+).
  - `hotfix/<description>`: For urgent fixes.
- **Commits**: 
  - Format: `<type>: <short description>` (max 50 chars).
  - Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`, `ci`.
  - Example: `feat: add user authentication`, `fix: resolve login redirect`, `docs: update API reference`.
  - Body: Wrap at 72 characters, explain what and why, not how.
  - Footer: References to issues (e.g., `Fixes #123`).

## Branch Strategy
- `main` branch is protected; direct pushes require pull request approval.
- All new work must be in a feature branch.
- Branches are deleted after merge (both locally and remotely).
- Release branches are not used; we rely on tags and GitHub releases.

## Commit Strategy
- Commit early and often.
- Each commit should be a single logical change.
- Avoid large, sweeping commits; break down work.
- Use `git commit --fixup` and `git rebase -i --autosquash` for cleanup.
- Before pushing, squash commits that are part of the same logical change if they are noisy.

## Pull Request Strategy
- PRs must target `main` unless otherwise specified (e.g., for a hotfix branch).
- PR title must be descriptive and follow the commit convention.
- PR description must include:
  - Summary of changes.
  - Related issue (e.g., `Fixes #123`).
  - Checklist of what was done (testing, documentation, etc.).
  - Any breaking changes or deprecations.
  - Screenshots or screen recordings for UI changes.
- PRs must be reviewed by at least one other maintainer (or AI acting as reviewer).
- All checks (CI, tests, lint) must pass before merge.
- Use the squash and merge strategy to keep a clean linear history.

## Review Strategy
- Reviewers must check:
  - Code correctness and adherence to engineering standards.
  - Documentation updates (if applicable).
  - Tests (new or updated).
  - No leftover debugging code or console.logs.
  - Proper error handling.
  - Security considerations.
  - Performance implications.
  - Backward compatibility.
  - Adherence to the project's Constitution and DNA.
- Reviews are asynchronous but timely (within 24 hours for non-urgent).
- Use GitHub's review system: comment, approve, or request changes.
- After addressing feedback, request a re-review.

## Documentation Rules
- All user-facing features must have documentation in `/docs` or in the respective package's `README.md`.
- API documentation must be kept up-to-date with code changes.
- Architectural decisions must be recorded in `ARCHITECTURE.md` or similar.
- Every public function, class, or module must have JSDoc comments.
- Documentation must be written in clear, concise English.
- When in doubt, over-document.

## Obsidian Synchronization Rules
- The `/brain` directory is the source of truth for institutional memory.
- The `/knowledge/agent-sessions` directory contains logs from AI agent sessions.
- These directories are synchronized with the user's Obsidian vault via the Obsidian Sync protocol (see `docs/obsidian-sync-protocol.md`).
- No files in `/brain` or `/knowledge/agent-sessions` should be generated by automated processes without human review.
- All Markdown files in these directories must follow the frontmatter standard (see `knowledge/agent-sessions/2026-07-11-hermes-phase-0-1-technical-skeleton.md` as an example).

## Zero Knowledge Loss Policy
- No important decision, design, or discovery may exist only in transient memory (chat, agent session, etc.).
- After any significant interaction (design decision, architecture discussion, etc.), the AI must update:
  - `LLM_HANDOFF.md` (for immediate handoff to the next AI).
  - `brain/BRAIN_LOG.md` (for institutional memory).
  - The relevant documentation file (if the decision affects architecture, process, etc.).
  - The Obsidian vault (via the knowledge sync protocol).
- Humans must also adhere to this policy when interacting with the AI.

## Definition of Done (DoD)
A task is considered done when:
- Code is written and meets engineering standards.
- Tests are added (if applicable) and pass.
- Documentation is updated (code comments, external docs, Obsidian).
- The changes are committed and pushed.
- A pull request has been created (if not already).
- The PR has been approved and merged.
- The changes have been deployed to the relevant environment (if applicable).
- The task is closed in the issue tracker.

## Definition of Ready (DoR)
A task is ready to be worked on when:
- It is clearly described in the issue tracker.
- Acceptance criteria are defined.
- Dependencies are identified and resolved or noted.
- The task is estimated and prioritized.
- Any required designs or specifications are available.
